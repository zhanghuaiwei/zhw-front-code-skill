## 🧠 核心心智模型

> **范式的灵魂**：所有真正能执行的规则，背后都有踩坑的故事

| 思维维度 | 核心信念与决策逻辑 |
|----------|-------------------|
| **🎯 第一仲裁原则** | **官方示例优先于任何人的个人习惯** —— 所有争议到此为止 |
| **架构分层观** | **吃过亏所以信邪**。components 边界破过防，所以一定要守住：<br>- `src/components/` → 🔒 无业务纯UI组件<br>- `src/views/components/` → 📦 可复用业务组件 |
| **类型安全执念** | **Java 式的类型严谨**。像写 Java 一样写 TS，每个变量、函数都要有明确类型。<br>知道紧急情况都会写 `any` → **必须在编译层面禁止**，靠自觉没用 |
| **反教条工程观** | 删除所有没用的"最佳实践"：<br>- 导入顺序没用 → 不要求<br>- 每个都 try-catch 太蠢 → 全局兜底就行 |
| **协作方法论** | **用机器代替人吵架**。能做成 ESLint 规则的就做成规则，做不成的再靠自觉 |
| **诚实原则** | 承认人性弱点。赶时间可能会偷偷违反写 any、不写注释 —— 但 400 行组件必须拆分，这是原则问题 |

## 📚 编码规范体系

### 2.1 代码格式化标准

| 规则项 | 规范值 | 严格程度 | 决策背景 |
|--------|--------|---------|---------|
| 缩进 | 2 个空格，禁止 tab | 🔴 强制 | 团队统一标准 |
| 分号 | 不需要分号 | 🟡 推荐 | **遵循 Vue3 官方编码风格** |
| 引号 | JS/TS 用单引号，HTML/CSS 用双引号 | 🔴 强制 | |
| 导入顺序 | **不做强制要求** | ⚪ 取消 | **实际不影响代码质量，删除教条** |
| 强制校验 | ESLint + TypeScript 严格模式 | 🔴 强制 | 编译层面拦截 |

### 2.2 命名规范体系

| 元素类型 | 命名风格 | 严格程度 |
|----------|---------|---------|
| 变量 / 普通函数 | 小驼峰（camelCase） | 🔴 强制 |
| 类 / 组件 / 接口 | 大驼峰（PascalCase） | 🔴 强制 |
| 常量 / 枚举值 | 全大写下划线（UPPER_CASE） | 🔴 强制 |
| 目录 / 非组件文件 | 小写中线分隔（kebab-case） | 🔴 强制 |
| React/Vue 组件文件 | 大驼峰（PascalCase） | 🔴 强制 |

**🔴 命名红线（零容忍）：**
- ❌ **绝不允许拼音命名**，哪怕是所有人都懂的业务黑话
- ❌ 绝不允许中英文混合命名

### 2.3 TypeScript 类型规范

1. **Java 式类型严谨**：所有函数必须声明返回类型，禁止依赖自动推导
2. **编译层面禁止 `any`**：业务代码使用 `any` 直接报编译错误
3. **紧急逃生舱**：可配置开关临时跳过类型校验（仅供紧急需求）

> 💡 不接受反驳：自行推导不符合类型规范安全，我要尽可能像 Java 一样，每个变量、方法都能找到对应的类型。

### 2.4 组件拆分原则

| 标准 | 要求 | 严格程度 |
|------|------|---------|
| 硬性红线 | 组件代码行数 **大于 400 行必须拆分** | 🔴 强制 |
| 推荐标准 | 单个组件理想情况不超过 200 行 | 🟡 推荐 |
| Props | 不超过 5 个，过多时考虑拆分 | 🟡 推荐 |
| 必须解构 Props | 禁止 `props.xxx` 写法 | 🔴 强制 |
| 逻辑分离 | JSX 只与视图相关，业务逻辑抽离到 Hooks | 🔴 强制 |

## 🛠️ 技术栈选型标准

## 💻 标准代码示例

### 4.1 React 标准组件模板

```tsx
interface UserCardProps {
  user: User
  onClick?: () => void
}

const UserCard: React.FC<UserCardProps> = ({
  user,
  onClick
}) => {
  // 1. State 定义
  const [isHover, setIsHover] = useState(false)

  // 2. Derived Value 计算
  const fullName = `${user.firstName} ${user.lastName}`

  // 3. Handler 定义
  const handleClick = (): void => {
    onClick?.()
  }

  // 4. Render
  return (
    <div className="user-card" onClick={handleClick}>
      {user.avatar && <Avatar src={user.avatar} />}
      <span>{fullName}</span>
    </div>
  )
}

export default UserCard
```

## 🏗️ 工程化最佳实践

### 5.1 标准目录结构

```
src/
├── components/             # 🔒 无业务纯UI组件
├── views/
│   ├── components/         # 📦 当前模块的业务组件
│   └── xxx/
│       ├── composables/    # 🧩 当前页面自治逻辑
│       ├── api.ts          # 📡 当前页面 API
│       └── index.vue       # 📄 页面入口
├── api/                    # 🔌 全局 API 封装
├── stores/                 # 📦 全局状态
├── utils/                  # 🛠️ 工具函数
└── assets/                 # 🎨 静态资源
```

### 5.2 测试集成规范

1. 每个前端项目必须集成测试框架
2. 覆盖范围：工具函数、业务组件、核心业务逻辑
3. 发布门禁：每次打包上线必须运行测试，通过才能打包
4. 可配置开关：紧急情况下可临时跳过

## ⚠️ 避坑指南

| 坑点 | 避坑方案 |
|------|---------|
| ❌ 组件过度臃肿（>400行） | 🔴 必须拆分，原则问题绝不放水 |
| ❌ 类型标注缺失 | 编译层面禁止 any |
| ❌ 拼音命名 | Code Review 零容忍 |
| ❌ 到处 try-catch | 统一全局错误拦截 |
| ❌ 跨层级依赖混乱 | 严格遵守单向依赖原则 |
