## Git 工作流

采用简化版 Git Flow：

| 分支 | 用途 |
|------|------|
| `main` / `master` | 生产环境代码，只能从 develop 合并 |
| `develop` | 开发环境主分支 |
| `feature/xxx` | 功能开发分支，从 develop 拉出 |
| `hotfix/xxx` | 紧急修复分支，从 main 拉出 |

**分支命名规范**：
- `feature/user-module` - 功能开发
- `hotfix/login-bug` - 紧急修复

## CI/CD 规范

✅ **提交代码自动触发**：

```
提交代码
    ↓
ESLint + StyleLint 检查
    ↓
TypeScript 类型检查
    ↓
✅ 单元测试运行 🔴 【发布门禁】
    ↓
构建
```

### 发布门禁核心规则

1. **测试不通过绝对不能打包上线**
2. 提供紧急开关，特殊情况可临时跳过
3. 目标是：不安全代码（人写的/AI写的）不能上线

> 💡 这个时代，AI 写的代码越来越多，测试闸门太重要了。

## 代码提交规范

采用 Conventional Commits 规范：

```
<type>(<scope>): <subject>
```

### Type 类型

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档更新 |
| `style` | 代码风格调整 |
| `refactor` | 重构 |
| `test` | 测试相关 |
| `chore` | 构建/工具相关 |

### 示例

```
feat(user): add user list page
fix(login): resolve token expired issue
docs: update readme
```

✅ **强制校验工具**：commitlint + husky

## 发布流程规范

### 开发环境
- 自动部署：develop 分支更新自动发布到测试环境

### 生产环境
- 手动触发，需要审核
- 发布前必须跑完整测试套件
- 发布失败自动回滚

### 版本号规范

```
主版本号.次版本号.修订号
  │       │       └──  bug 修复，向下兼容
  │       └──────────  新增功能，向下兼容
  └──────────────────  不兼容的重大改动
```

示例：`1.2.0`, `1.2.1`, `2.0.0`
